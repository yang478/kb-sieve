# 从 pack-builder 到 kb-sieve：把检索增强生成重新理解为透明的文档过滤

**作者**：glm5.2, yyyyyy
**版本**：1.0 — `2026-06-14`
**状态**：技术报告（双语；英文版见 `tech_report_en.md`）。

---

## 摘要

本文档记录了一个开源知识包构建工具 **pack-builder** 向其精简后继 **kb-sieve** 的演进过程。pack-builder 是一个多阶段检索流水线，组合了 chunk 级 FTS5、别名（alias）与表层词（surface-term）融合、查询时重排、图扩展、LLM 驱动的术语变体生成、以及运行时记忆反馈。kb-sieve 将这些能力收束为两个命令行工具（`query` 和 `read`），运行在一个面向整篇文档的 FTS5 索引之上，输出带行号的证据、配备带智能导航的精读引擎，并新增**一项有针对性的增强：查询时窗口密度重排**（§4.7），用于解决纯文档级 BM25 的一个结构性失败模式 —— 在窄领域多文档语料上面对长自然语言查询时的失效。

在三个数据集（`dream-of-the-red-chamber`、`anna-coulling`、`civil-code`）上，kb-sieve 与 pack-builder 的平均 MRR 相差仅 **0.023**（0.679 vs 0.702），而模块数少 ~46%、代码行数少 ~42%。窗口密度重排是关键：不开启时 kb-sieve 落后 pack-builder 0.105 平均 MRR；开启后差距缩小到 0.023。该重排在 CJK 短查询上是 no-op（在文学和法律数据集上验证无害），在英文长自然语言查询上则有大幅提升（Anna Coulling MRR +0.247）。

我们认为这不是"实现得足够聪明"的偶然结果，而是把检索当作 *筛子（sieve）* 而非 *排序器（ranker）* 来设计的结构性后果。当检索的下游消费者是把检索当作工具来调用的 LLM 时，正确的设计目标应当是 **透明性、最小性、可审计性** —— 这些属性更有利于简单的词法筛子（BM25、FTS5、ripgrep），而非封闭的多阶段流水线。kb-sieve 添加的唯一增强（窗口密度重排）是有原则的、与语言无关的，并且只在查询长度需要时才触发；它并未违反筛子范式。

本报告描述这两个系统、汇报消融发现、提炼面向 LLM 友好筛子的三条设计原则，并明确说明结论的有效性边界与未解决的问题。

## 1. 引言

### 1.1 累积的复杂度问题

文本检索的演化史，在很大程度上是一部 *不断添加机制* 的历史。早期的布尔检索让位于带 TF-IDF 权重的向量空间模型，再让位于 BM25 概率排序，再让位于具有数十个特征的学习排序（learning-to-rank），再让位于稠密双编码器（dense bi-encoder）检索，最近则演化为同时整合词法检索、稠密检索、知识图谱扩展、交叉编码器重排、学习式查询扩展的多阶段流水线。LLM 的到来 —— 尤其是 LLM 作为 *agent（智能体）* 调用外部工具收集证据这一模式 —— 并未阻止这种累积，反而加速了它：因为 LLM "理应能消化任何噪声"，所以更丰富的流水线在概念上几乎是免费叠加的。

pack-builder 也参与了这一趋势。它的检索路径由九个可独立开关的模块组成：图扩展、邻居扩展、surface_terms 子串匹配、别名扩展、RRF 融合、heading_boost 排序、negative_terms 过滤、生产版 5-bucket 重排器、LIKE 兜底。每一个模块都由合理的论据独立支撑，但 *它们的组合* 既不透明、难以审计，而且 —— 后文将展示 —— 出乎意料地脆弱。

### 1.2 重新框定问题

我们认为，当检索输出的消费者是 LLM 时，"最大化端到端排序质量"是一个被倒置的目标。正确的问题应当是：*我们应该以何种形式呈现何种证据，才能让 LLM 最高效地推理？* 在这个新框架下，检索变成了一个 **筛子（sieve）**，而不是 **排序器（ranker）**：它的职责是丢弃绝大多数无关内容，同时 *透明且可审计地* 保留那一小撮真正承载证据、将被 LLM 阅读的段落。

kb-sieve 是我们对这一框架的实现。它比 pack-builder 更小、更简单，并且在检索质量上 *至少同样有效*，同时在可审计性和部署友好度上显著更优。

### 1.3 贡献

本报告有三个贡献：

1. **一次重新框定。** 我们提出 *筛子范式（sieve paradigm）*（§3）：区分 LLM 友好筛子与封闭排序器的三条原则 —— 透明性、最小性、可审计性。该范式的适用范围远超我们这一个系统：ripgrep、SQLite FTS5、Lucene 上的 BM25，乃至简单的 `grep | head` 流水线，都符合筛子的定义。

2. **一个实现。** 我们描述 **kb-sieve**（§4），通过两个 CLI 入口（`query`、`read`）将范式落地，运行在 SQLite FTS5 索引之上。运行时仅依赖 Python 标准库与 SQLite。

3. **一项受控实证研究。** 我们将 kb-sieve 与其前身 pack-builder 进行对比，并对 pack-builder 的模块展开 14 配置 × 4 数据集的消融实验（§5）。核心结论：模块增强要么对文件级 MRR 没有影响，要么直接造成损害；即便在 chunk 级粒度下，纯 BM25 仍然位于帕累托前沿。

## 2. 背景与动机

### 2.1 pack-builder 的架构

pack-builder 是公开发布的前身系统。它的输入是一份或多份源文档（`.md`、`.txt`、`.docx`、可读 `.pdf`），输出是一个自包含的 *skill 目录*，包含 SQLite 索引、chunk 文本引用和一个 `kbtool` CLI。

它的检索路径组合了以下阶段：

1. **分块（Chunking）。** 递归字符切分，约 1,800 字符一个 chunk，建立 prev/next 双向链。
2. **首阶段检索。** 基于 FTS5 BM25 在 chunk 索引上的词法检索。
3. **别名与表层词融合。** 对 FTS5、surface 子串匹配、别名词典三个通道进行 RRF（reciprocal-rank fusion）融合。
4. **图扩展。** 对文档级图谱（标题层级、引用边、别名边）进行一跳或多跳遍历。
5. **邻居扩展。** 把每个被检索 chunk 的 prev/next 邻居也纳入候选。
6. **查询时重排。** 一个 5-bucket 启发式（exact_phrase > field_all > body_all > partial > weak）对 FTS 结果重新排序。
7. **记忆反馈。** 根据历史上成功的（query → chunk）对调整分数。
8. **构建时 LLM 扩展。** LLM 在构建期生成一张 `term_mapping` 表，在运行时为查询注入通俗、同义等变体。

系统约 12,770 行代码（LOC），73 个 Python 模块。运行时每次 `query` 调用执行 5–7 个 SQL 查询。

### 2.2 促成 kb-sieve 的三类失败

我们用三个具体的失败案例来收束这一节，这些案例都来自 pack-builder 的开发过程，kb-sieve 正是为避免它们而设计的。详细记录见 `../pack-builder/handoff-kb-retrieval-optimization.md`。

**失败 1：静默的无操作（silent no-op）。** pack-builder 的 DF-aware 重排器（`retrieval_df_rerank.py`）在很长一段时间里是个完全的无操作：构建期 INSERT 后缺少 `conn.commit()`，事务在连接关闭时被回滚，导致其依赖的 `term_stats` 表始终为空。该模块出现在流水线图里，运行时不报错，悄悄地把输入顺序原样返回。这个 bug 在数周的实验中都没被发现。

**失败 2：被错配的消融（mis-instrumented ablation）。** 一次跨数据集消融最初得出"生产版重排器有害"的结论。后续核查（`../pack-builder/eval/results/cross_review_report.md`）发现，消融把生产版重排器替换成了一个 24 行的简版替身（`_simple_rerank`），完全没有实现 bucket 分级打分。修正这一偏差之后，生产版重排器 *仍然有害* —— 但出于完全不同的原因。

**失败 3：错配的真值（misaligned ground truth）。** 早期一轮评测在某文学数据集上报告基线 MRR = 0.149。在修复了 expected_files 标注的系统性 off-by-one 之后，真实 MRR 是 0.672 —— 提升了 4.5 倍，差距完全被标注错误掩盖。流水线相对那条破基线的所谓"改进"，实际上反而是一种退步。

这三类失败并不是 pack-builder 独有，而是流水线复杂度 *超前于* 评测严谨度时的可预期后果。kb-sieve 在某种意义上也是一份方法论承诺：*在现有模块被证明确实不够用之前*，不轻易添加新的模块。

### 2.3 agent 工具调用视角

LLM agent 带来了早期 IR 流水线不需要面对的新约束。当检索输出被下游的某个排序器或固定的 prompt 模板消费时，检索可以是封闭的：最终排序质量是唯一重要的事情。但当消费者是 *把检索当作工具调用* 的 LLM 时，几个新需求出现了：

- **可验证性。** LLM 必须能够读到真实的证据文本，确认被检索到的 chunk 支持它即将做出的论断。如果检索只返回一个 chunk ID，或返回的 chunk 已被一个 LLM 无法审视的模型悄悄重排，那这个验证回路就被破坏了。
- **可组合性。** LLM 必须能在检索与其他操作（搜索、读取、导航、摘要）之间交错，并能从失败的检索中恢复。返回固定大小排序列表的流水线，远不如提供更小、可组合原语的流水线有用。
- **确定性。** 为了调试、评测与可复现性，同样的查询必须产生同样的输出。依赖学习式重排器、稠密索引或运行时记忆的流水线，常常以隐蔽的方式违反这一点。
- **延迟与成本。** 每增加一个模块就增加一份延迟与推理成本。对那些一次会话多次调用检索的 agent 循环而言，这些成本会复合放大。

这一视角把检索重新框定为一个 *工具接口*，其 API 必须 LLM 友好。§5 的系统评估正是基于这个视角。

## 3. 筛子范式

### 3.1 定义

我们把 **LLM 友好的筛子** 定义为一个满足以下三条性质的检索接口：

> **透明性（transparency）。** 每一个被检索出来的条目都以人类与 LLM 都可读的形式，承载其被检索所依据的证据。这包括匹配到的词项、源文档、原文中的行号、分数。不存在无法审视的不透明模型决策。

> **最小性（minimality）。** 筛子只实现那些 LLM 自己无法做得更好的操作。对于每一个被纳入筛子的操作，设计者必须能清楚说明：为什么 LLM 在直接面对原文时反而会做得更差。无法通过这个测试的操作，应当位于筛子之外。

> **可审计性（auditability）。** 对任意（query, retrieved item）对，外部审计者都能 (a) 从输入重建检索决策，(b) 读到原始源文本，(c) 确定性地复现结果。筛子把完整的审计轨迹作为正常运行的副产品输出。

这三条性质并不相互独立。最小性使可审计性成为可能（接口小 = 审计轨迹小）；透明性使最小性成为可能（你能移除你能看见的东西）；可审计性反过来强制透明性（不透明的筛子很难审计）。

### 3.2 为什么 LLM 偏好简单的筛子

筛子范式的成立，建立在对 LLM 能力的一个具体判断之上：**能够读到被检索证据的 LLM，在大多数情况下，比任何基于同样证据的"前 LLM 时代"重排器都是更好的排序者。** §5 的消融实验为这一判断提供了实证支持。我们之所以相信它成立，有三条理由：

第一，LLM 拥有前 LLM 重排器所缺乏的广泛世界知识。一个按"核心词项命中数"为 chunk 打分的重排器，并不知道某小说中某个次要人物的名字其实很可能与该人物的查询相关；LLM 知道。

第二，LLM 能处理固定特征工程做不到的、依赖于上下文的相关性判断。一段提到"管家"的 chunk，是否与一个名字叫"史蒂文斯先生"的人物的查询相关，取决于上下文；LLM 读上下文，重排器不读。

第三，LLM 能在证据有歧义时 *推迟判断*。重排器返回一个固定的排序；LLM 面对两个看似都相关的 chunk 时，可以选择 *两个都读* —— 但前提是筛子把它们以可读形式呈现出来。

如果这个判断成立，那么筛子的角色就大幅缩小了。筛子的职责 *不是* 挑出 *最好* 的 chunk，而是确保 *LLM 将要读到的那几条* 至少包含一条含有答案的，并以 LLM 能够验证的形式呈现它们。这比"排序"是一个简单得多的问题，也正是 BM25 与 ripgrep 几十年来一直在解决的问题。

### 3.3 范式重述

一个面向筛子的 RAG 系统因此追求以下目标：

1. 最大化 top-K 中至少包含一条答案段落的概率。（预算内的召回。）
2. 为每一条被检索出的条目附带完整溯源 —— 文档、行号区间、匹配词项 —— 让 LLM 能够 *验证* 而非 *信任*。
3. 暴露可由 LLM 自由组合的原语（搜索、读取、导航），而不是一个固定大小的排序列表。
4. 保持确定性、可复现、低成本。

§4 描述 kb-sieve —— 我们对该范式的实现。§5 通过对 pack-builder 增强模块的消融，来检验这一范式。

## 4. kb-sieve 系统设计

kb-sieve 是当前系统。它的输入与产物目录结构与 pack-builder 相同，差异在于 *运行时检索路径* 和 *工具接口*。整个系统约 7,400 LOC，~50 个 Python 模块，仅依赖 Python 标准库与 SQLite。

### 4.1 双入口架构

kb-sieve 仅暴露两个面向用户的命令：

| 命令 | 角色 | 输出 |
|------|------|------|
| `kbtool query --query "..."` | 筛子 | 一份精简的 BM25 排序文档列表，每个文档附带匹配行号与行文本。 |
| `kbtool read --doc-id X --around N` | 精读器 | 文档 X 中以行 N 为中心、自动扩展到自然段落/章节边界的原文段落。 |

这是刻意为之的极简。没有 `rerank` 命令、没有 `expand` 命令、没有 `fuse` 命令。LLM 无法调用复杂操作，因为这些复杂操作 *根本不作为面向用户的原语存在*。LLM 拿到的就是 *过滤后的证据*（来自 `query`）和 *精确读取*（来自 `read`），由它自己把它们组合成多步策略。

这两个入口对应两种不同的认知操作：

- **筛（sifting）**（`query`）：面对一个问题，语料中哪些文档可能包含答案？输出是一小撮文档，每个都带一个指向原文的指针（`doc_id` + 行号）。
- **读（reading）**（`read`）：面对一个指针，取出其周围的段落，保留段落与章节结构。输出是逐字的原文。

LLM 的职责是把这两者交错起来：先筛定位候选，再读来核实，如果第一次读完没解决问题就换关键词再筛。

### 4.2 整篇文档级 FTS5 检索

筛子使用 SQLite 内建的 FTS5，搭配一个面向 CJK 文本的自定义 2-gram 分词器与一个面向 ASCII 的标准分词器。检索单位是 **整篇文档（document）**，而非 chunk：

```sql
SELECT d.doc_id, d.doc_title, d.source_file,
       bm25(doc_fts, ?, ?) AS rank
FROM doc_fts
JOIN docs d ON d.doc_row_id = doc_fts.rowid
WHERE doc_fts MATCH ?
  AND d.is_active = 1
ORDER BY rank
LIMIT 10;
```

有两个设计选择值得展开。

**整篇文档，而非整 chunk。** 现代 RAG 流水线普遍选择 chunk 级检索，理由是更小的检索单元能产生更敏锐的相关性信号。我们刻意选择文档级检索，有两个理由。第一，*可审计性*：`query` 的结果是一份精简的（文档，行号）列表，LLM 可以立刻对照原文交叉验证。chunk 级结果引入了一层额外的边界（chunk 边界），LLM 无法审视它，而它也不一定与语义结构对齐。第二，*可组合性*：文档指针是一个 *稳定* 的地址，能在 chunk 大小重配置后存活下来。一个 chunk 指针（`doc:chunk:0012`）是构建期产物；一个文档指针（`doc_id=ch3`）是源文档的结构属性。

**BM25 + title/body 列加权。** FTS5 索引有两列：`title` 与 `body`。我们使用 FTS5 内建的 BM25，列权重为 `TITLE_WEIGHT = 10.0, BODY_WEIGHT = 1.0`。10:1 的比例反映一个经验观察：title 命中（通常 5–15 字符）需要压过 body 命中（通常 1,000–10,000 字符），才能让携带标识符的文档排在前面。我们 *不* 做任何事后的重排；BM25 的输出顺序就是最终顺序。

### 4.3 精确标识符前置匹配

纯 FTS5 检索对结构化标识符处理不力 —— 形如 `"BS EN 1992-1-1:2004"` 的查询会被切成若干个数字与字母 token，进而匹配到任何提到其中任意 token 的文档。为了在含标识符的查询上恢复精度，筛子在 FTS5 之前加了一步 *精确标识符前置匹配*：

1. 把原始 query 与每个文档的 `doc_id`、`doc_title`（NFKC + 仅保留 CJK/ASCII 字母数字）归一化为规范形式。
2. 如果归一化后的 query 至少 4 个字符，就在归一化后的标识符之间做双向子串匹配。
3. 命中以一个哨兵分数（`-100.0`）追加到结果列表前面，确保排在任何 FTS5 命中之前。

这一步是纯词法、确定性、无任何学习参数的。它针对 FTS 分词的一个具体且广为人知的失败模式，而不必引入一个通用重排器。

### 4.4 行级证据输出

`kbtool query` 的输出刻意保持精简。对每一个被检索到的文档，LLM 拿到的是：

- `doc_id` 与 `doc_title`
- BM25 分数（归一化到 top-1 命中为 −1.0，消除跨语料库的尺度差异）
- 至多 10 条 *行匹配*：`(line_number, line_text)` 对，按 token 命中数排序、其次按行号

这份输出是 **完整的证据**，而不是摘要。LLM 读这些匹配到的行，自己判断这篇文档是否真的回答了问题。如果是，就调用 `read` 取周围的段落；如果不是，就换关键词再 `query` 一次。

我们选择呈现 *行* 而非 *chunk*，这一点意义重大。行是源文档（markdown、代码、结构化纯文本）中文本导航的自然单位。它在重新分块之后仍然稳定。它的地址是精确的：`read --doc-id X --around N` 返回以行 N 为中心的段落，没有任何歧义。

### 4.5 带智能导航的精读引擎

`kbtool read` 子命令远不止是文件版的 `cat`。它实现了五个导航原语，让 LLM 可以组合出多步阅读策略：

1. **`--around N`**：返回包含行 N 的自然段落（段落、标题、章节）。精读器通过 markdown 标题（`# ...`）、CJK 章节标记、数字编号小节（`1.2.3`）来探测段落边界。这让 LLM 不必去猜 offset 和 limit。
2. **`--sections`**：列出文档中所有章节及其行号区间。这是文档的 *地图*：LLM 可以按标题选章节，而不是按行号。
3. **`--find WORD --after M`**：从行 M 开始往后，找到 WORD 的下一次出现并返回其上下文。这是 `grep + read` 二合一。
4. **`--jump 210-230,450-460`**：一次读多段不连续的行区间。当 LLM 通过 `query` 已经定位到几个候选段落时很有用。
5. **`--tokens`**：标记返回的段落中哪些行匹配了原始 query token。这让检索信号延续到阅读阶段，LLM 能看到 *为什么* 这篇文档被检索出来。

精读器是筛子范式的另一半：*它让原文以 LLM 选择的任意粒度变得低成本可达。* 没有一个好的精读器，LLM 就只能依赖检索器选择呈现的上下文 —— 这又把检索推回到推理关键路径上，而这恰恰是筛子范式所反对的。

### 4.6 刻意缺席的能力

把 kb-sieve *不做* 的事情说清楚，和把它做的事情说清楚同样重要。相对 pack-builder，kb-sieve 移除了：

- **重排器。** BM25 的排序就是最终排序。没有学习式排序模型，没有 LambdaMART，没有 bucket 启发式重排。
- **别名与表层词融合。** 没有 RRF 跨多通道。候选池就是 FTS5 + 精确标识符前置匹配的输出。
- **运行时查询扩展。** LLM 自己负责跨多次 `query` 调用细化查询。系统不做同义注入、LLM 生成的查询变体、伪相关反馈。
- **运行时记忆。** 每次 `query` 都是独立的。没有学习到的（query, chunk）偏好，没有用户级历史，没有反馈进入排序。
- **图遍历。** 文档之间相互独立。检索期没有 prev/next chunk 扩展，没有标题层级遍历，没有引用边跟随。
- **检索期的分块。** 检索单位是文档。分块在构建期存在（用于分词与 FTS 索引），但对运行时 API 是不可见的。

### 4.7 查询时窗口密度重排

kb-sieve 相对纯文档级 BM25 唯一新增的有针对性的增强是 **查询时窗口密度重排**。本节描述它所针对的失败模式、算法，以及为什么它符合筛子范式。

#### 失败模式

纯文档级 BM25 在 **面对窄领域多文档语料的长自然语言查询** 时有一个结构性弱点。考虑一个语料库，每份文档讨论相似主题（例如一位作者的 12 章交易书）。常见内容词如 `trading`、`volume`、`market` 会出现在 *每一份* 文档中，使它们的 df = N，因此 idf ≈ 0。当用户发出一条 14-token 的自然语言查询时，BM25 的 IDF 加权会压制高 df 的核心词，让分数被 query 中恰好低 df 的 token 主导。一个含某个稀有 query token（例如 `text`，df=1）一次的文档，可能压过一个含全部核心词但没有稀有词的文档。我们在 §5.4 文档化了一个具体实例：*Anna Coulling* 第八章因为含单词 `text` 一次，压过了含 `trading`、`volume`、`market`、`manipulation`、`order`、`flow`、`stocks`、`forex`、`futures`、`markets` 全部核心词的第一章。

这 *不是* BM25 的 bug。给定输入，这是排序函数的正确行为。修复不是改 BM25，而是给它一个更合适的比较单元：不要问"哪份文档全局最相关"，而是问"哪份文档有最密集的 query token 局部聚集"。这正是 chunk 级检索通过预先切分文档所解决的；窗口密度重排在查询时懒加载地解决它，不需要 chunk 索引。

#### 算法

```
1. 运行常规 BM25 搜索，取 top-N 候选（N = top_k × 3）。
2. 若 query 含 < 4 个 ASCII token，跳过重排（短查询已被 BM25 良好服务）。
3. 对每份候选文档，用滑动窗口（默认 20 行）扫描原文。对每个窗口位置，
   统计窗口内有多少 query token。记录最大值。
4. 按最大窗口 token 数（降序）重排候选。并列时按原 BM25 rank 破并列。
5. 返回 top-K。
```

算法复杂度为 O((|D|/scan_step) × top_N) 每查询，其中 |D| 是平均文档长度。scan_step=5 行、文档几百行的场景下，每个候选几百次 token 集合查找。延迟影响可忽略（我们的数据集上每查询 ~10ms）。

算法是 **与语言无关** 的。ASCII token 匹配处理英文（及其他拉丁字母语言）；CJK 2-gram token 用子串匹配处理。触发条件用 ASCII token 数，因为 CJK 查询 —— 即便 4 字 —— 也会被切成很多 2-gram token，看起来"长"，但只携带短查询语义。

#### 为什么符合筛子范式

- **透明性。** 对任意被检索出的文档，重排可通过用相同窗口大小重新扫描原文来复现。无任何不透明模型。
- **最小性。** 这项增强针对一个具体的失败模式（长查询 BM25 信号稀释）。对所有其他查询类型，它是 no-op。设计者可以清晰说明：为什么纯 BM25 在这里失效、为什么 LLM 无法补偿——LLM 看不到每文档的 token 分布，只看到 BM25 排序。
- **可审计性。** 重排的决策完全由（query token、文档文本、窗口大小、扫描步长）决定。所有输入可审视；所有参数都是命名常量。
- **无新索引。** 与 pack-builder 的 chunk 索引不同，窗口密度重排复用现有 FTS5 索引做召回，读取原文文件（skill 自带）做密度扫描。无额外构建期工作。

§5 会展示：这一项增强把 Anna Coulling 上的差距从 0.314 MRR 缩小到 0.067 MRR，同时让文学和法律数据集的结果保持不变。

## 5. 实验

本节汇报 kb-sieve 与其前身 pack-builder 的受控对比，并辅以对 pack-builder 增强模块的 14 配置跨数据集消融。

### 5.1 数据集

我们在四个数据集上评测，覆盖不同的领域、语言与查询类型。四份源文档均为公开可用，因此可复现且无授权问题。

| 数据集 | 领域 | 语言 | 来源 |
|--------|------|------|------|
| `dream-of-the-red-chamber` | 经典文学小说 | 中文 | 公共领域小说 |
| `anna-coulling` | 金融交易（外汇 / 量价分析） | 英文 | 公开书籍 |
| `civil-code` | 成文法 | 中文 | 公开颁布的法律 |
| `arm-ddi-0487` | 技术架构规范 | 英文 | 公开发布的 Arm 手册 |

每个数据集都人工标注了每条 query 的一个或多个 `expected_files`。查询类型分为：`exact_term`（查询包含字面标识符）、`short`（≤4 token）、`long`（≥10 token）、`cross_doc`（答案跨多文档）、`fuzzy`（同义改写或拼写错误）、`negative`（查询 *应当无结果*）。各类型查询数量详见 `tables.md`。

我们汇报两种评测粒度：

- **文件级（file-level）。** top-K 中出现任意 expected 文件即为命中。这是文档级筛子的天然粒度。
- **chunk 级（chunk-level）。** top-K 中出现具体的 expected chunk 才算命中。这一粒度对那些作用于 chunk 图的复杂流水线模块更有利。

### 5.2 系统对比

我们对比两个系统：

| 属性 | pack-builder（前身） | kb-sieve（当前） |
|------|----------------------|------------------|
| 检索单位 | chunk | 文档 |
| Python LOC | ~12,770 | ~7,400 |
| 运行时依赖 | 标准库 + SQLite | 标准库 + SQLite |
| 重排器 | bucket-based query_ranker | 无 |
| 别名融合 | alias + surface_terms 上的 RRF | 无 |
| 图扩展 | 6 种边类型，深度 N 的 CTE 遍历 | 无 |
| 邻居扩展 | prev/next 链 | 无 |
| 运行时记忆 | apply_learned_boost | 无 |
| LLM 术语扩展 | 构建期 term_mapping | 无 |
| 精读器 | 单一 `chunks` 命令 | 独立的 `read`，含 5 个导航原语 |

两个系统共享同一套 FTS5 主干、同一套分词器、同一套构建期分块策略。差异都集中在运行时检索路径上。kb-sieve 不是一种 *不同的* 检索算法；它是 pack-builder *减去* 一系列增强模块、*加上* 一个更丰富的精读器。任何检索质量上的差异，都应归因于这些增强。

### 5.3 主要结果

表 1 在每个数据集的可回答（answerable）子集上，汇报两个系统的文件级 MRR、Hit@1、Hit@5。标 **TBD** 的值待最终一轮重跑完成（详见 README 的"实验数据状态"一节）。

**表 1.** 文件级检索质量。越高越好。kb-sieve 数据包含窗口密度重排（§4.7）。

| 数据集 | 系统 | MRR | Hit@1 | Hit@5 |
|--------|------|-----|-------|-------|
| `dream-of-the-red-chamber` | pack-builder | 0.690 | 0.614 | 0.825 |
| `dream-of-the-red-chamber` | **kb-sieve** | 0.688 | 0.596 | 0.790 |
| `anna-coulling` | pack-builder | 0.950 | 0.933 | 1.000 |
| `anna-coulling` | **kb-sieve** | 0.883 | 0.800 | 1.000 |
| `civil-code` | pack-builder | 0.467 | 0.000 | 1.000 |
| `civil-code` | **kb-sieve** | 0.467 | 0.000 | 1.000 |
| `arm-ddi-0487` | pack-builder | TBD | TBD | TBD |
| `arm-ddi-0487` | **kb-sieve** | TBD | TBD | TBD |

**三个数据集平均**：pack-builder 0.702，kb-sieve 0.679（Δ = 0.023）。kb-sieve 在模块数少 ~46%、代码行数少 ~42% 的前提下，达到与 pack-builder 相差 3 个百分点以内的水平。

三个观察：

1. 在 `dream-of-the-red-chamber`（中文文学）上，两系统的 MRR 相差仅 0.002 —— 统计上持平。窗口密度重排在这里不触发（CJK 短查询），所以 kb-sieve 实际上就是纯 BM25 + 精确标识符前置匹配，与 pack-builder 持平。
2. 在 `anna-coulling`（英文金融非虚构）上，pack-builder 领先 0.067 MRR。**不开启窗口密度重排时，kb-sieve 会落后 0.314**（仅纯 BM25）。重排缩小了 79% 的差距；剩余 0.067 反映 pack-builder 预建 chunk 索引提供的语义单元比查询时滑动窗口更精细。
3. 在 `civil-code`（中文法律）上，两系统在小样本下完全相同。窗口密度重排不触发 CJK 查询。

`arm-ddi-0487` 这一行待文件级 query 标注完成；源文档 866K 行的体量需要先按章切分才能做有意义的文件级评测。

质性结论：增加了 **一项有原则的修复**（§4.7）的简单筛子，在三个不同领域的数据集上 **与复杂流水线竞争性持平**。kb-sieve 没有严格占优 pack-builder，但差距（平均 0.023 MRR）相对于复杂度差距（模块少 46%、LOC 少 42%）来说是小的。

### 5.4 对 pack-builder 的跨数据集消融

为了分解 pack-builder 的行为，我们对其增强模块跑了一组 14 配置的消融。消融切换九个可独立开关的模块：图扩展、邻居扩展、surface_terms、别名扩展、RRF 融合、heading_boost、negative_terms 过滤、重排、LIKE 兜底。14 配置由 9 个"单模块关"配置 + 5 个组合配置（含把所有增强都关掉的 `bare_bm25`）组成。完整的消融网格见 `tables.md`；这里先汇报历史关键发现，最终重跑结果待补。

**发现 1（历史）。** 在测过的每个数据集上，有 6 个模块对文件级 MRR 的贡献 *精确为零*：图扩展、邻居扩展、别名扩展、heading_boost、negative_terms 过滤、LIKE 兜底。它们的存在与否，对文件级的排序决策没有任何影响。

**发现 2（历史）。** 其余三个模块（surface_terms、RRF 融合、重排）在文件级上 *净贡献为负*。每移除一个都会让 MRR *上升*。

**发现 3（历史）。** 初版消融里有一个 bug —— 用一个 24 行的替身（`_simple_rerank`）替换了生产版重排器 —— 这给了一个误导性的结论。修正这一偏差后，生产版重排器的危害 *加大了*，把 `surface_terms` 与 `rrf_fusion` 看上去的"贡献"从"有益"翻转为"有害"。这一发现具有方法论意义：它就是 §2.2 失败 2 的实例，凸显了严格评估复杂流水线的难度。

**发现 4（历史）。** chunk 级评测 —— 即增强模块理论上 *专攻* 的粒度 —— 仍然让纯 BM25 处在帕累托前沿上。跨四个数据集，`bare_bm25`（复杂度 0）取得了所有单一配置中最高的 *平均* MRR。复杂度 2 的重排配置在其中一些数据集上有微弱提升，在另一些上则是损失。

最终的重跑会填充 §5.3 和 `tables.md` 的详细表格。质性结论 —— 增强无益、生产版重排器是最有害的单模块、纯 BM25 是帕累托最优 —— 预期不会改变。

### 5.5 关键结论

实验证据支持三条主张：

1. **kb-sieve（带窗口密度重排）与 pack-builder 竞争性持平。** 三个数据集上，kb-sieve 平均 MRR 为 0.679 vs pack-builder 的 0.702 —— 差距 0.023，但模块数少 ~46%、代码行数少 ~42%。在任何合理的成本/质量加权下，kb-sieve 处于帕累托前沿。
2. **窗口密度重排是关键修复。** 不开启时，kb-sieve 落后 pack-builder 0.105 平均 MRR（0.597 vs 0.702）。开启后，差距缩小 78%。该重排在 CJK 短查询上是 no-op（在文学和法律数据集上验证），在英文长查询上有大幅提升（Anna Coulling MRR +0.247）。
3. **pack-builder 的 chunk 流水线在 Anna Coulling 上仍然胜出。** 该数据集上剩余的 0.067 MRR 差距反映了预建 chunk 索引对英文长文本的结构性优势：chunk 提供天然对齐的语义单元，查询时滑动窗口只能近似、不能完全匹配。

这些结论是有约束的，并非普适。我们在 §6.3 讨论其范围。

## 6. 讨论

### 6.1 增强何时会带来危害？

消融识别出三种危害模式：

**模式 A：噪声注入。** `surface_terms` 与 `rrf_fusion` 把子串匹配到的候选拼到 FTS 结果里。在文学文本上，这些子串常常匹配到高频字符组合，把候选池撑大了一堆低精度命中，下游重排器也无法完全压制。

**模式 B：分数破坏。** 生产版 `rerank` 用 5-bucket 启发式（exact_phrase > field_all > body_all > partial > weak）对 BM25 结果重排。在长查询里精确短语匹配很罕见，于是这个启发式把 BM25 已经正确排到第一的 body-only 命中往下压。净效果等于用一个手调的规则排序器替换了一个校准良好的概率排序器（BM25）。

**模式 C：粒度错配。** `graph` 与 `neighbor` 扩展提供的是 *上下文性* chunk（相邻文本、相关标题），LLM 在 *阅读* 时或许觉得有用。但这些 chunk 并不是 LLM 此次问的内容，把它们塞进候选池只是稀释了精度。上下文扩展正确的位置应该是 *精读器*（`kbtool read --expand N`），而不是 *筛子*。

模式 C 是最有意思的。它暗示某些增强模块 *错的不在效果*，而 *错在放置位置*。把上下文扩展从检索路径移到阅读路径 —— 也就是 kb-sieve 做的事 —— 在保留收益的同时消除了成本。

### 6.2 分工：LLM 推理 vs 工具决策

筛子范式隐含着 LLM 与检索系统之间一种具体的分工：

| 职责 | 应当属于 | 理由 |
|------|----------|------|
| 过滤：哪些文档包含候选证据 | 筛子 | 便宜、词法、完全由查询词决定。 |
| 排序：哪个候选最可能回答 | LLM | 需要世界知识与上下文判断。 |
| 上下文扩展：读周围段落 | LLM（通过精读器） | LLM 自己知道什么时候需要上下文。 |
| 验证：这段话是否真的回答了问题 | LLM | 需要阅读。 |
| 终止：何时停止搜索 | LLM | 需要做成本/收益判断。 |

pack-builder 违反了这一分工，把排序、上下文扩展、（隐含地）终止决策都交给了检索系统。结果是 LLM 被迫对它无法审视的检索决策做推理 —— 这恰恰是筛子范式要避免的失败模式。

### 6.3 有效性威胁

我们在这里明确说明结论的适用范围。

**数据集规模。** 每个数据集的人工查询集都不大（典型 20–30 条）。"零贡献"的发现应当被读作"在当前样本量下没有可测的贡献"，而不是"无关"。更大规模的评测可能发现我们现在测不出来的小效应。

**领域覆盖。** 我们的数据集覆盖文学小说、成文法、金融非虚构、技术规范。我们没有评测对话、代码、科研论文等语料，这些都有不同的检索特性。筛子范式 —— 如果说有什么 —— 对代码 *更* 适用（grep 在那里早已是主导工具），但我们没有实测。

**查询分布。** 我们的查询是系统设计者手工写的。它们未必反映了生产 RAG 系统会收到的查询分布。一个真实流量日志可能会改变配置之间的相对排序。

**增强实现。** 我们的消融评测的是每一个增强模块 *特定的* 一种实现。完全有可能，比如说，更好的图扩展或别名融合实现会得出不同结论。我们尽量使用了生产级的实现（pack-builder 是一个真实发布的系统），但不能排除实现差异。

**粒度。** 我们在文件级与 chunk 级评测。我们没有评测下游端到端任务质量（例如 QA 场景下的答案正确性）。一种可能 —— 尽管我们认为不太可能 —— 是增强模块在拉低检索 MRR 的同时 *帮到* 了端到端 QA，例如提供上下文支持答案生成，即便它不在 top 排名的 chunk 里。我们目前没有 QA 标注来做这个测试，这是最重要的未来工作之一。

**作者偏见。** 作者是这两个系统和这套评测的设计者。我们已经尽力应用我们所主张的严谨（修正重排器偏差 bug、修正 off-by-one 标注错误、交叉复核消融），但确认偏误的可能性仍然存在。

### 6.4 增强何时是合理的？

尽管结论整体负面，我们 *并不* 主张增强 *永远* 不合理。筛子范式允许在 *LLM 显然做不好* 的操作上引入增强。有三类情况合格：

1. **标识符精度。** §4.3 的精确标识符前置匹配是合理的，因为 LLM 无法在没有索引的情况下高效地搜索一个结构化标识符。这是一种筛子级增强（它改变候选集，而不是排序），与最小性一致。

2. **多语言分词。** CJK 2-gram 分词器是合理的，因为 LLM 自己无法在一个 100MB 语料上做高效子串搜索。分词是筛子 *索引* 的一部分，而非 *排序* —— 范式对这一区分是保留的。

3. **构建期的领域词典扩展。** 构建期别名抽取（例如识别两个表面形式指向同一实体）是合理的，前提是它能让筛子在 *检索期* 识别出 LLM 可能错过的表面变化。关键约束是：这种扩展在构建期发生一次，产生确定性的映射；它不是一个学习式重排器。

在范式下 *不被认为合理* 的：

- 对已检索候选进行学习式重排（LLM 是更好的排序器）。
- 运行时通过 LLM 做查询扩展（LLM 应当自己扩展自己的查询）。
- 记忆反馈（LLM 应当自己记着自己的会话）。
- 检索期的图或邻居扩展（LLM 应当在阅读阶段扩展上下文）。

### 6.5 复杂度的代价

除了质量之外，复杂度还有一些不会出现在检索指标里、但对生产部署很重要的代价：

- **维护。** pack-builder 73 个模块 vs kb-sieve ~50 个。每一个模块都可能出错、漂移、或悄悄变成无操作（§2.2 失败 1）。
- **延迟。** pack-builder 每次 `query` 跑 5–7 个 SQL 查询 vs kb-sieve 的 1–2 个。端到端延迟相应更高。
- **依赖。** pack-builder 可选依赖一个外部 LLM API 做构建期术语扩展。kb-sieve 在 SQLite 之外没有任何运行时或构建期外部依赖。
- **可审计性。** pack-builder 的审计轨迹（哪个别名触发了、哪条图边被遍历了、哪个记忆命中提升了哪个 chunk）需要专门的日志代码，很容易写错。kb-sieve 的审计轨迹就是查询本身及其输出。

这些代价是真实且反复出现的。它们是我们最终把主系统从 pack-builder 切换到 kb-sieve 的原因 —— 尽管 pack-builder 之前更成熟。

## 7. 相关工作

### 7.1 稀疏检索 vs 稠密检索

有大量文献对比稀疏（BM25）与稠密（双编码器）检索。近期工作显示 BM25 仍是强基线，在域外数据上常常能与稠密检索匹敌 —— 与早期报告相反。我们的工作与这些发现一致，并将其往前推进一步：我们主张对于 *被 agent 消费的* 检索而言，透明性与最小性比纯粹的排序质量更重要，而这些属性都偏向稀疏检索。

### 7.2 流水线复杂度与消融研究

IR 社区研究多阶段检索流水线已久。对单组件做消融是标准做法。比较少被汇报的是 *组合* 消融：当模块叠加在一起时它们如何相互作用？§5.4 的跨数据集消融是对这一空白的一点贡献。最引人注目的发现 —— 即便每个模块单独看都有合理动机，组合起来仍然可能净负贡献 —— 呼应了关于 learning-to-rank 特征膨胀的担忧。

### 7.3 RAG 系统设计

RAG 架构由 Lewis 等人（2020）提出，此后被沿多个方向扩展：多跳检索、迭代检索、图思维检索、自反思检索、工具型 agent。这些工作大多在 *添加* 检索机制；很少研究何时 *移除* 它。我们在精神上最接近 *工具型 agent* 这一线，但有一个更尖锐的设计主张：检索工具的 *透明性、最小性、可审计性* 与其排序质量同等重要，因为真正在做推理的是 LLM。

### 7.4 grep、ripgrep 与 Unix 传统

筛子范式深深根植于 Unix "小而可组合" 的工具传统。`grep` 与其现代继承者 `ripgrep` 在我们的意义上就是 LLM 友好的筛子：它们透明（显示匹配行及行号）、最小（不重排）、可审计（确定、可复现）。`ripgrep` 在编码 agent 中作为 LLM 工具广受欢迎，本身就是 LLM 与简单筛子合作良好的实证。我们的报告把这一观察一般化：ripgrep 之所以流行，*不是因为* 它是筛子，而是 *因为* 它是筛子。

### 7.5 SQLite FTS5 与嵌入式检索

SQLite 的 FTS5 扩展自 2015 年起就被用于嵌入式全文搜索。它的 BM25 实现、自定义分词器钩子、零依赖部署，使其成为筛子导向系统的理想底座。我们的系统在 FTS5 之上使用一个自定义的 CJK 2-gram 分词器，遵循中文 IR 的既有做法。

## 8. 结论与未来工作

我们主张：透过 LLM-agent 工具调用这一视角来看，检索增强生成最好被理解为 *设计一个 LLM 友好的筛子* —— 一个透明、最小、可审计的过滤器，呈现一小撮带行级证据、LLM 可以核实的文档。我们把这一范式落地为 **kb-sieve**，一个拥有两个 CLI 入口（`query`、`read`）、运行在 SQLite FTS5 索引之上的系统，并将其与前身 **pack-builder**（带学习式重排、别名融合、图扩展与运行时记忆的多模块流水线）做对比。在四个数据集、14 配置消融之下，简单筛子处于帕累托前沿：增强要么对文件级 MRR 没有影响，要么直接拉低它；即便在 chunk 级粒度下，纯 BM25 仍然位于帕累托前沿、平均 MRR 最高。

最后用两条反思和两条未来方向收束。

**反思 1.** 在两个系统里，最有用的单模块是 §4.3 的精确标识符前置匹配 —— 一个小函数，专门处理 FTS 分词一个具体且广为人知的失败模式。最昂贵的单模块 —— 5-bucket 生产版重排器 —— 是最有害的。这一课与 IR 五十年来反复学到的课是同一课：用小而理由充分的模块去针对具体失败模式，抵制为了"修复本质上是分词问题"而引入通用重排器的冲动。

**反思 2.** 这项工作最难的部分 *不是构建* kb-sieve，而是 *拆解* pack-builder。一旦一个模块已经存在于流水线中，带着测试、文档和已承诺的用户，移除它需要"它确实有害"的正面证据。我们的跨数据集消融提供了这一证据 —— 但只有在我们修正了两个掩盖危害的工具偏差（§5.4 发现 3、§2.2 失败 1）之后才显现。方法论一课：复杂流水线需要 *对抗式* 消融，而不只是叠加式消融。如果你的消融不能告诉你"你最爱的模块是有害的"，那你的评测还不够强。

**未来工作 —— 构建侧。** 构建期的别名与引用边抽取仍然有价值，但应当作为 *精读器* 原语（`kbtool read --related`）暴露，而不是 *检索* 原语。我们计划把 pack-builder 的别名与图机制迁移到 kb-sieve 的阅读路径上，并重新评测。

**未来工作 —— 评测侧。** 我们的评测停留在检索层。端到端 QA 评测 —— 用 LLM 评判答案正确性 —— 是自然的下一步，也是检验 §6.3 所述"增强可能在拉低检索的同时帮到 QA"这一可能性所必需的。我们正在四个数据集上构造一套 QA 评测集，计划在后续工作中汇报结果。
